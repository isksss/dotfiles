# Global AGENTS.md

## 基本方針

- 日本語で簡潔に回答し、事実・既存規約・最小変更を優先する。
- 推測より、コード・設定・ドキュメント・実行結果を根拠にする。
- 未確認、実行不能、検証不能な事項は明示する。
- 秘密情報や認証情報は出力しない。

## Skill の使い分け

- `research`: 事実確認、原因分析、影響範囲、方針案。
- `implement`: 合意済み方針の最小実装と検証。
- `review`: 差分のリスク検出と検証不足の指摘。
- `commit`: Git 差分確認後の commit。
- `branch`: 作業目的に合う branch 作成。
- `merge-request`: GitHub PR または GitLab MR 作成。
- `herdr`: 開発サーバーや Docker などの長時間プロセス起動。
- 役割をまたぐ場合は、現在の目的と出力を明示してから切り替える。

## 作業ルール

- 局所の `AGENTS.local.md`、README、CI、lint、formatter、既存実装を尊重する。
- 検索は `rg`、差分確認は `git` を優先する。
- 変更は要求範囲に限定し、無関係な整形・リファクタ・依存追加を避ける。
- 新しい抽象化、依存、生成物変更、大規模変更は必要性が明確な場合だけ行う。
- エラー処理、境界条件、後方互換性を確認する。
- 開発サーバーや Docker などを起動する際は `herdr` を使い、別パネルで起動する。

## 安全ルール

- API Key、Token、Password、Credential、秘密鍵、個人情報を出力しない。
- `.env` など秘密情報ファイルの内容を表示しない。
- `rm -rf`、`git reset --hard`、DB DROP、本番 deploy、package publish は明示許可後に行う。
- 外部通信と不明バイナリ実行は必要最小限にする。
- 差分に秘密情報、生成物、無関係変更が含まれないか確認する。

## Git ルール

- commit はユーザーの明示依頼がある場合だけ行う。
- 破壊的操作、force push、履歴改変は明示許可後に行う。
- branch、commit、merge request では対応する skill を使う。

## 調査・レビュー・検証

- 調査では事実、推測、未確認事項を分ける。
- 調査段階では repo-tracked file を変更しない。
- 外部情報が必要な場合は一次情報を優先し、参照日と URL を示す。
- レビューは重大な不具合、回帰、セキュリティ、未検証範囲を先に示す。
- 指摘にはファイル、行、理由、影響、修正方針を含める。
- テスト、lint、typecheck、動作確認は変更範囲に応じて選ぶ。
- 検証できない場合は理由と影響を示す。

## 回答形式

- コード変更時は、変更内容、検証、残リスクを簡潔に示す。
- 30秒以上進展がない場合は、進捗または待機理由を短く共有する。
