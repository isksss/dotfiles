# grill-me workflow

## 事前調査

質問前に必ず調査する。

対象:

- ソースコード
- テスト
- README
- docs
- ADR
- CI/CD
- 設定ファイル
- DBスキーマ
- API定義

コードベースから判明する内容は質問禁止。代わりに報告する。

報告形式:

```markdown
リポジトリ調査結果:

- ...

次に確認したい事項:

...
```

## ヒアリング方針

- 一度に質問する内容は1件のみ。
- 複数質問禁止。
- 回答後に次へ進む。
- 実装、コード生成、Issue生成へ進まない。
- 設計ツリーの分岐をたどり、意思決定間の依存関係を順に解決する。
- 固定チェックリストを順番に消化するのではなく、現在の論点から最も重要な未決定事項を1つ選ぶ。
- 「幅」より「深さ」を優先する。新しい洞察が得られなくなるまで同じテーマを掘り下げる。
- 甘い回答、曖昧な返答、既存決定との矛盾には追加質問で突っ込む。
- 既に決まった内容と新しい回答の整合性を毎回確認する。

## 確認観点

以下は順番固定ではなく、抜け漏れ確認の観点として使う。

- 目的: 何を実現したいか、なぜ必要か、成功条件。
- 利用者: 誰が使うか、利用頻度、利用場面。
- 業務フロー: 開始条件、操作手順、完了条件。
- ドメイン: エンティティ、属性、関係性、ライフサイクル。
- API: 入力、出力、エラー、制約。
- データ: 保存先、更新ルール、削除ルール、履歴管理。
- 権限: 実行可能者、閲覧可能者、管理者権限。
- セキュリティ: 認証、認可、入力検証、監査ログ。
- 障害時動作: 失敗条件、リトライ、ロールバック、ユーザー通知。
- 運用: ログ、メトリクス、アラート。
- テスト: Unit Test、Integration Test、E2E Test。
- リリース: Migration、後方互換、Feature Flag、Rollout。

## 質問フォーマット

以下形式固定。

```markdown
### Q<番号>: <質問文>

<なぜこの質問が重要か>

- A: <選択肢>
- B: <選択肢>
- C: <選択肢>

**推奨: <A/B/C>** — <理由>
```

## 終了判断

- 主要な設計分岐をたどり終えた。
- 決定間の依存関係に矛盾がない。
- 未確認事項が実装判断を左右しない。
- ユーザーが追加深掘りを求めない。

## 仕様完成条件

- 要件曖昧さなし
- 業務フロー明確
- データモデル明確
- API明確
- 権限明確
- テスト観点明確
- リリース方針明確
- 受入条件明確

## 完了時出力

```markdown
## まとめ

### 決まったこと

- ...

### 実装者が守る基準

- ...

### ADR候補

- Context: ...
- Decision: ...
- Consequences: ...

### 受入条件

- ...

### リスク

- ...

### 未解決事項

なし

仕様確定。実装へ進む？
```

## 注意

ADR候補は出力するだけに留め、ファイル作成や更新はしない。
