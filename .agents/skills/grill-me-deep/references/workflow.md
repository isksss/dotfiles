# grill-me-deep workflow

## 事前調査

質問前に必ず調査する。

対象:

- ソースコード
- テスト
- README
- docs
- ADR
- CI/CD
- 設定ファイル
- DBスキーマ
- API定義
- 監視、ログ、アラート設定
- リリース、migration、feature flag 関連設定

コードベースから判明する内容は質問禁止。代わりに報告する。
既存ADRがあれば必ず読み、今回の意思決定と衝突しないか確認する。

## ヒアリング方針

- 一度に質問する内容は1件のみ。
- 複数質問禁止。
- 回答後に次へ進む。
- 大型開発で実装者が判断に迷う点を残さない。
- 実装、コード生成、Issue生成へ進まない。
- 設計ツリーの分岐をたどり、意思決定間の依存関係を順に解決する。
- 固定チェックリストを順番に消化するのではなく、現在の論点から最も重要な未決定事項を1つ選ぶ。
- 「幅」より「深さ」を優先する。新しい洞察が得られなくなるまで同じテーマを掘り下げる。
- 甘い回答、曖昧な返答、既存決定との矛盾には追加質問で突っ込む。
- 回答ごとに、既存ADRや既に決まった内容との整合性を確認する。

## 確認観点

以下は順番固定ではなく、抜け漏れ確認の観点として使う。

- 目的: 何を実現したいか、なぜ必要か、事業・利用上の成功条件。
- 利用者: 誰が使うか、利用頻度、利用場面、影響範囲。
- 業務フロー: 開始条件、操作手順、分岐、完了条件、例外。
- ドメイン: エンティティ、属性、関係性、ライフサイクル、不変条件。
- API: 入力、出力、エラー、制約、互換性。
- データ: 保存先、更新ルール、削除ルール、履歴管理、migration。
- 権限: 実行可能者、閲覧可能者、管理者権限、委譲。
- セキュリティ: 認証、認可、入力検証、監査ログ、機密情報。
- 障害時動作: 失敗条件、リトライ、ロールバック、冪等性、ユーザー通知。
- 運用: ログ、メトリクス、アラート、ダッシュボード、調査手順。
- テスト: Unit Test、Integration Test、E2E Test、移行検証、回帰観点。
- リリース: Migration、後方互換、Feature Flag、段階 rollout、rollback。

## 質問フォーマット

```markdown
### Q<番号>: <質問文>

<なぜこの質問が重要か>

- A: <選択肢>
- B: <選択肢>
- C: <選択肢>
- D: <選択肢>

**推奨: <A/B/C/D>** — <理由>
```

## ADR分類

既存ADRへの影響がある場合、以下の分類案を出す。ファイル作成や更新はしない。

- 新規追加: 新しい意思決定を記録する。
- 更新: 技術スタック、データモデル、ステータス、公開APIなど意思決定を変更する。
- 上書き: typo、表現調整、決定を変えない説明追記。

## 終了判断

- 主要な設計分岐をたどり終えた。
- 決定間の依存関係に矛盾がない。
- 既存ADRとの整合性が確認できた。
- 未確認事項が実装、運用、リリース判断を左右しない。
- ユーザーが追加深掘りを求めない。

## 仕様完成条件

- 要件曖昧さなし
- 業務フロー明確
- データモデル明確
- API明確
- 権限明確
- セキュリティ方針明確
- 障害時動作明確
- 運用監視方針明確
- テスト観点明確
- リリース、移行、rollback 方針明確
- 受入条件明確

ADR候補と分類案は出力するだけに留め、ファイル作成や更新はしない。
