# develop workflow

1. `research` として局所ルール、既存差分、関連実装、検証方法を確認する。計画や診断だけの依頼なら変更せず結果を返す。
2. 要求が明確なら `implement` として必要なファイルだけを変更する。安全な仮定で進められない曖昧さは確認する。
3. 変更範囲に応じた test、lint、typecheck、構文検証、動作確認を実行する。
4. `review` として差分を確認する。重大な問題があれば実装へ戻り、修正後に再検証する。
5. 変更内容、検証、残リスクを簡潔に報告する。

branch、commit、push、PR/MR、作業記録はユーザーが明示した場合だけ対応する skill へ渡す。長時間プロセスは `herdr` を使う。
