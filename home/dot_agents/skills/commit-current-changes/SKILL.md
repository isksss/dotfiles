# commit-current-changes

このスキルは、現在の Git 変更を確認し、簡潔な prefix 付き日本語コミットメッセージで意図の明確なコミットにするために使う。

## 手順

1. ステージ前にリポジトリの状態を確認する。
   - `git status --short` を実行する。
   - `git diff --stat` と `git diff --cached --stat` を実行する。
   - 必要な差分を `git diff` と `git diff --cached` で読む。
   - 未追跡ファイルは、ステージ前にファイル名と内容を確認する。

2. コミット範囲を決める。
   - 既定では現在の全変更を確認する。
   - 生成物、キャッシュ、依存関係、ログ、ローカル環境ファイル、秘密情報は除外する。
   - `.env`、`.env.*`、`node_modules`、`.nuxt`、`.output`、`.nitro`、`.cache`、`dist` などのローカル生成物は含めない。
   - 無関係な変更が混ざっている場合、実用的なら分割コミットを優先する。分割しない場合は、まとめる範囲を説明する。
   - ユーザーが明示的に依頼しない限り、ユーザーの変更を revert したり破棄したりしない。

3. prefix 付きの日本語コミットメッセージを作成する。
   - 1行目は `[prefix] 日本語の要約` の形式にする。
   - prefix は変更内容に応じて `add`、`fix`、`update`、`remove`、`refactor`、`docs`、`test`、`chore` から選ぶ。
   - `[add]`: 新機能、新ファイル、新設定の追加。
   - `[fix]`: 不具合修正。
   - `[update]`: 既存機能、設定、文言の更新。
   - `[remove]`: 不要なコードやファイルの削除。
   - `[refactor]`: 挙動を変えない整理。
   - `[docs]`: ドキュメントのみの変更。
   - `[test]`: テストの追加や修正。
   - `[chore]`: ビルド、依存関係、開発補助などの保守作業。
   - 1行目は、ユーザーに見える成果または保守上の成果を簡潔に要約する。
   - `更新` や `修正` のような曖昧な要約だけで終わらせない。
   - 主な対象と操作を含める。例: `[update] Nuxt MCP の設定 URL を更新`、`[add] コミット作成用の Codex skill を追加`。
   - 差分に補足説明が必要な場合だけ本文を追加する。

4. ステージしてコミットする。
   - 選択したファイルだけをステージする。
   - サンドボックスの制約で Git メタデータが読み取り専用となり `git add` や `git commit` が失敗した場合、必要な承認付きで同じコマンドを再実行する。
   - `git` コマンドが読み取り専用メタデータ以外の理由で失敗した場合は、エラー内容をユーザーに報告し、可能な解決策を提示する。
   - 選んだ prefix 付き日本語メッセージでコミットする。
   - `Co-authored-by`は、ユーザーが明示的に依頼した場合以外は使用しない。

5. 確認して報告する。
   - `git status --short` を実行する。
   - `git log -1 --oneline` を実行する。
   - コミットハッシュとメッセージを日本語で報告する。
   - 未コミットの変更が残る場合は、何がなぜ残っているかを明確に伝える。

## 安全確認

- コミット前に、認証情報、トークン、秘密鍵、個人情報、マシン固有のパスが差分に含まれていないか確認する。
- 秘密情報や private なファイルの可能性があるものを見つけた場合は、作業を止めてユーザーに進め方を確認する。
- lint や typecheck などの確認が関連し、短時間で実行できる場合はコミット前に実行する。実行しない場合は、その旨を伝える。
