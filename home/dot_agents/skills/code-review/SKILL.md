# コードレビュースキル

このスキルは、プロジェクト固有のコーディング規約・品質ルールおよび一般的なソフトウェアエンジニアリング知識に基づいて、コード変更をレビューし構造化レビューレポートを生成します。

> **重要**: このスキルは**読み取り専用**である。ファイルの作成・編集・削除は一切行わず、レビュー結果のレポート出力に専念すること。

## レビュー方針

- まず差分を確認し、次に変更に関係する周辺ファイルを読む。いきなり広い範囲を探索せず、差分に直接関係する箇所から入る。
- 変更の意図を推測で補わず、差分・既存実装・プロジェクト指示から確認できる事実だけで判断する。
- 影響の大きい問題を先に報告する。仕様逸脱、データ破壊、セキュリティ、破壊的変更、既存利用者への影響を優先する。
- 問題が見つからない場合も、確認した観点と未確認の前提を簡潔に残す。

## レビュー実行手順

### 1. 差分を把握する

- `git diff --stat` と `git diff --cached --stat` で変更の全体像を確認する。
- `git diff` と `git diff --cached` で実際の差分を読む。
- 未追跡ファイルがある場合は、必要に応じて内容を確認し、レビュー対象に含めるか判断する。

### 2. プロジェクト固有のルールや品質基準を確認する

- 下記ファイルを確認し、プロジェクト固有のコーディング規約や品質ルールを把握する。
  - `AGENTS.md`
  - `.github/copilot-instructions.md`
  - `.github/instructions/**/*.instructions.md`
- 変更対象の周辺に `README.md`、設定ファイル、テスト、型定義、公開 API の利用箇所がある場合は、差分と矛盾しないかを確認する。
- プロジェクト固有の命名規則、レイヤー分離、エラーハンドリング、ログ出力、フォーマット規約がある場合は、その規約に照らして読む。

### 3. 差分を各観点でレビューする

- 変更された行だけでなく、その行が依存している前提条件も確認する。
- 1 つの変更が複数の観点にまたがる場合は、最も重大な観点を優先して記録する。
- 変更意図が明示されている場合は、その意図を満たしているかを確認し、意図が不明な場合は安全側に倒して評価する。

#### 差分レビュー

- 追加・削除・リネーム・設定変更・依存追加など、変更の種類を分けて確認する。
- 既存の規約、既存の命名、既存のエラーパターン、既存のフォーマットとずれていないかを見る。
- 明らかな重複、不要な複雑化、冗長な分岐、不要な副作用がないかを確認する。

#### 一般的なバグ検出

- null / undefined / 空配列 / 空文字 / 0 / 境界値 / 並行実行 / 再試行 / 例外経路を確認する。
- 条件分岐が抜けていないか、戻り値やエラー処理が既存の呼び出し側と整合しているかを確認する。
- 状態更新、キャッシュ、非同期処理、順序依存、初期化順の変更によって壊れる箇所がないかを確認する。
- セキュリティに影響する入力処理、権限、認証、データ露出、シェル実行、パス操作は特に厳しく見る。

#### ファイル全体整合性チェック

- 変更箇所だけでなく、同じファイル内の既存ロジックと矛盾していないかを確認する。
- そのファイルにあるヘルパー、型、定数、コメント、既存の設計方針と食い違う変更がないかを見る。
- 変更によって同じファイル内の別の分岐や後続処理が成立しなくならないかを確認する。

#### 影響範囲分析

- 公開関数、クラス、設定項目、CLI 引数、環境変数、テンプレート、エクスポート、ファイル名変更などの破壊的変更を確認する。
- 利用側が想定している引数、戻り値、型、例外、デフォルト値、イベント順序が変わっていないかを確認する。
- 影響が大きい変更は、呼び出し側や関連テストも見て、後方互換性の有無を明確にする。

#### 要件適合性チェック

- 差分が、ユーザー要件、issue、PR 説明、設計意図に一致しているかを確認する。
- 要件の一部しか満たしていない、あるいは別の挙動を新たに混入させている場合は問題として報告する。
- 要件が曖昧な場合は、どの前提が未確認かを明記する。

### 4. 安全確認を行う

- 差分に認証情報、トークン、秘密鍵、個人情報、機密情報、マシン固有のパス、ローカル環境依存の値が含まれていないか確認する。
- 生成物、キャッシュ、ログ、依存関係のロックファイル以外の不要な大量変更、意図しないフォーマット崩れが混入していないか確認する。
- 破壊的な削除、広範囲のリネーム、設定の上書き、権限緩和などの高リスク変更を見つけた場合は、レビュー結果で明示する。
- 秘密情報や private なファイルの可能性があるものを見つけた場合は、レビューを続ける前にユーザーへ確認を求める。
- lint、typecheck、テスト結果がレビュー判断に有効で、短時間で確認できる場合は参照する。参照できない場合は、その制約を明記する。

### 5. レビューレポートを生成する

- 問題は、見つけた順ではなく重要度の高い順にまとめる。
- 各項目には、できるだけ具体的なファイル名と行番号を付ける。
- 行番号が特定できない場合は、その理由を簡潔に添える。
- 重大な問題がない場合でも、確認した観点と「重大な問題なし」を明記する。
- 問題の種類が複数にまたがるときは、最も本質的な分類に寄せて 1 回だけ報告する。

## レポートの構成

レビューレポートは以下の構成で出力すること:

```markdown
# レビューレポート

## ルール違反・品質問題

- {ファイル名}:{行番号} - {問題の内容}

## 一般的なバグ

- {ファイル名}:{行番号} - {問題の内容}

## ファイル全体の整合性問題

- {ファイル名}:{行番号} - {問題の内容}

## 影響範囲の問題

- {ファイル名}:{行番号} - {問題の内容}

## 要件適合性の問題

- {ファイル名}:{行番号} - {問題の内容}
```

## 出力ルール

- レポートは日本語で書く。
- 断定できる事実と推測を混ぜない。推測が必要な場合は、その前提を明示する。
- 修正案は、必要最小限で具体的に書く。大きな再設計案は、問題の指摘とは分けて扱う。
- 問題がない場合は、各分類ごとに「該当なし」と書いてよい。ただし、確認を省略した観点は省略したと分かるように書く。
